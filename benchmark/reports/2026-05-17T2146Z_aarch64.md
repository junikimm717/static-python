## Analysis

**What we did.** Added `-flto-partition=none` to both `CFLAGS` and
`LDFLAGS` in `configure-wrapper.sh` and rebuilt the static interpreter
only. Default gcc LTO partitioning is `=balanced`, which splits the
program into parallel chunks for faster link wallclock at the cost of
some cross-partition inlining decisions. `=none` keeps the whole
program in a single LTO partition for maximally aggressive
cross-procedure inlining at the cost of a single-threaded link.

The reason we hand-patched the flag instead of doing the cleaner
`./configure --with-lto` for libpython itself is documented inline in
`configure-wrapper.sh`: our musl-cross-make gcc is built
`--disable-shared` (it has to be -- we ship a fully-static interpreter
and we want a self-contained toolchain), so the linker plugin only
exists as `liblto_plugin.a`, not `.so`. Binutils ld can only `dlopen`
a `.so` plugin, so `-fuse-linker-plugin` (which `--with-lto` adds)
fails with `error: -fuse-linker-plugin is not supported in this
configuration`. CPython's `--with-lto` apparatus hardcodes that flag
and offers no way to opt out of it. So we replicate the *useful* knob
(`-flto-partition=none`) by hand and let the dynamic baseline keep
using upstream's machinery.

What the numbers say:

- **`listcomp` was the headline win: 1.03x -> 1.27x. Static absolute
  went 19.0 -> 15.6 ns, 18% faster.** This is a pure-libpython row (no
  C extension boundary), so all of the gain comes from the libpython
  internals being inlined more aggressively under `=none`. This
  *falsifies* the guess I floated in the 2112Z analysis -- that
  dynamic had a small libpython-internal LTO edge from `=none` that
  static was missing. The reality is the opposite: static was leaving
  significant inlining wins on the table by accepting default LTO
  partitioning. Now that it doesn't, it strictly dominates dynamic on
  every libpython-internal row.
- **`except_path` jumped 1.05x -> 1.19x** -- same reason (5.5% static
  improvement on a libpython-internal row), and consistent with what
  we'd expect since the C-level exception restore path involves a lot
  of small helpers that benefit from cross-partition inlining.
- **The C-extension-boundary rows also widened, but less dramatically.**
  `str_format` 1.15x -> 1.25x, `json_roundtrip` 1.16x -> 1.22x,
  `attr_access` 1.13x -> 1.17x, `fib_iter` 1.16x -> 1.09x (this one
  *narrowed*, dynamic happened to do well in run-to-run noise this time
  on this row). These rows were already favoring static thanks to the
  static binary's ability to LTO across the C-extension <-> libpython
  boundary; the new whole-program inlining adds to that lead but isn't
  the dominant effect on them.
- **`fib_recursive` is still 0.78x. Static went 21.6 -> 22.2 ns -- got
  *slightly* worse**, though that's within run-to-run noise. Three
  rounds of optimisation (PGO, then upstream `--with-lto` recipe via
  hand-replication of `=none`) have now had their shot at this row
  and not moved it. The "structural call-site bloat" hypothesis (LTO
  inlined `_PyEval_EvalFrameDefault` too aggressively on the recursive
  call path; M4 Pro's branch target predictor mispredicts at varying
  recursion depths as a result) keeps surviving every test we throw at
  it. `=none` should have made this *worse* if the hypothesis is right,
  since `=none` inlines even more aggressively than `=balanced`. The
  small absolute regression on static (21.6 -> 22.2 ns) is consistent
  with that direction.

### What the asymmetries breakdown from 2112Z looks like now

Re-stating the three asymmetries between static and the current
apk-linked dynamic baseline:

1. `_*.so` <-> `libpython3.13.so` boundary -- cannot LTO across.
   (only present on dynamic)
2. `_*.so` <-> Alpine `lib*.so.N` boundary -- cannot LTO across.
   (only present on dynamic)
3. Alpine `lib*.so.N` is not internally LTO'd at all.
   (only present on dynamic)

This run *also* added a fourth, smaller asymmetry now that we know
about it:

4. Static's libpython gets `-flto-partition=none` whole-program
   inlining across libpython + libssl + libsqlite + libz + libffi +
   liblzma + libbz2 + libncursesw + libuuid + libreadline. Dynamic's
   libpython gets `-flto-partition=none` *only across libpython
   itself*. The aggressive cross-partition inlining is now strictly
   wider in scope on static.

The 0.05 geomean delta from 2112Z to here -- that's asymmetry (4)
talking. Asymmetries (1)+(2)+(3) are still present and are still
unresolved by any of the changes we made today.

### What is missing now (revised priorities)

1. **`DYNAMIC_USE_INTREE=1` dynamic build** -- still #1 priority, now
   even more so. With static now reliably 1.11x faster geomean (and up
   to 1.27x on individual rows), the right next experiment is the
   apples-to-apples LTO-vs-LTO baseline. Build our `openssl` /
   `sqlite` / `zlib` / `libffi` / `xz` / `bz2` / `ncurses` / `libuuid`
   shared (with our `-O3 -flto -flto-partition=none` `CFLAGS`),
   install to `build-dyn-$(TARGET)/`, then point `dynamic-build.sh`
   at that prefix instead of `apk`. That isolates asymmetries (1) and
   (4) from (2)+(3): if the new "everything is LTO'd shared" dynamic
   binary still loses to static on the boundary rows, the residual win
   is exclusively the cross-`.so` cost -- exactly the structural
   "static linkage" win we want to be able to claim.
2. **`perf stat` on `fib_recursive` on real aarch64 hardware** -- now
   even higher confidence that this is a structural issue with our
   compiler decisions and not anything PGO+LTO can fix. Three rounds
   of optimisation tweaks have not moved 0.78x. Need branch-misses +
   iTLB-load-misses + L1-icache-load-misses on a tight `fib(20)` loop
   on both binaries on a host with a real PMU.
3. **`STATIC_NO_LTO=1` static build** -- mirror experiment for
   completeness; now expected to lose ~10% geomean vs current.
4. **musl libc LTO via musl-cross-make patches**; same as before.
5. **Cross-arch PGO via qemu-user**; same as before.
