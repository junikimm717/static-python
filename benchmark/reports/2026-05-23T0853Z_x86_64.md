## Analysis

Fresh AWS x86_64 run on an AMD EPYC 9R14 host. No interpreter recipe change
relative to the current PGO+LTO reports: static is still the 16.0 M
whole-program-LTO build, and the dynamic baseline was rebuilt before this run
with CPython's `--enable-optimizations --with-lto` path so the comparison is
not the confounded static-vs-system-only run.

The natural diff target is the Cascade Lake
[`2026-05-19T0223Z_x86_64.md`](./2026-05-19T0223Z_x86_64.md) report because it
is the latest x86_64 PGO+LTO report with the same binary sizes and dynamic
baseline recipe. The older Zen 5
[`2026-05-18T0000Z_x86_64.md`](./2026-05-18T0000Z_x86_64.md) report is useful
mainly for the `stdlib` startup outlier.

### CPU geomean moves the upper bound: 1.20x -> 1.27x

This is the strongest static-vs-dynamic CPU geomean so far. The cross-host
PGO+LTO band becomes N1 1.07x / M4 Pro 1.11x / Zen 5 1.18x / Cascade Lake
1.20x / AWS EPYC 1.27x. That falsifies the comfortable "about 7-20%" range in
the cliffs notes; it is now **7-27%** across hosts.

The move is not one row. Compared with Cascade Lake, EPYC is materially higher
on `fib_iter` (1.22x -> 1.48x), `arith_loop` (1.38x -> 1.51x), `str_format`
(1.20x -> 1.37x), `func_call` (1.32x -> 1.39x), and `except_path`
(1.23x -> 1.38x). `json_roundtrip` also sets a new high at 1.28x. The rows that
do not move much (`attr_access`, `listcomp`, `dictops`, `regex_match`) stay
inside the previous spread.

### `fib_recursive` remains the only CPU loss

`fib_recursive` lands at 0.93x, the least-bad value in the PGO+LTO set but
still a loss. The table now reads M4 Pro 0.78x / Cascade 0.85x / Zen 5 0.89x /
N1 0.92x / EPYC 0.93x. That keeps the N1 perf conclusion alive: the problem is
not a one-host frontend artefact but something about the eval-loop recursive
path under this static recipe. EPYC softens the magnitude; it does not change
the direction.

### Startup matches the stable cluster, not the Zen 5 outlier

Startup geomean is 1.10x, essentially Cascade Lake's 1.10x and inside the
M4/N1/Cascade cluster. The `stdlib` scenario is 1.13x, which is another point
against the Zen 5 0.89x `stdlib` result being a structural x86_64 problem.
Treat the Zen 5 result as a host/run outlier unless it reproduces.

### Next step

The highest-value experiment is still `DYNAMIC_USE_INTREE=1`: build the dynamic
baseline against shared versions of our pinned, LTO-built dependencies and
rerun on this EPYC host. The rows that widened most here (`str_format`,
`json_roundtrip`, `func_call`, `except_path`) are exactly where whole-program
scope and extension/library boundaries are the live explanation.

For the persistent negative row, run a targeted PMU probe if this AWS instance
exposes the counters:
