*almost*. Re-bench after two things happened on this tree:
(1) the `configure-wrapper.sh` LTO flag tweak in commit `6a73c5c`
("change configure pipeline") that landed on 2026-05-19, swapping
`-flto -flto-partition=none -fuse-linker-plugin -fno-fat-lto-objects`

static side, and (2) an unrelated build-tree corruption (mixed root /

deps-aarch64-linux-musl/Python-3.13.13 python-static-aarch64-linux-musl`
and a fresh in-container `make USE_CROSSMAKE=1 python3`. So this is
the first PGO+LTO aarch64 datapoint with the new wrapper flags, and
the natural diff target is
[`2026-05-17T2359Z_aarch64.md`](./2026-05-17T2359Z_aarch64.md) (same
host, old wrapper).

- Apk-installed `gcc` in the Alpine container got bumped between 2359Z

  consistent with codegen difference than load noise.

### Static binary size: 13.8M → 15.8M (+14.5%, +1.99 MB)

**This is the only material change in this report.** The recipe diff
between 2359Z (old `configure-wrapper.sh`) and this run (new wrapper)
on the static side is exactly:

```
- -flto -flto-partition=none -fuse-linker-plugin -fno-fat-lto-objects
+ -flto=auto -flto-partition=none -fuse-linker-plugin
```

Semantically those two should be almost identical:

- `-flto` → `-flto=auto`. With `-flto-partition=none` pinned in both
  cases, GCC has only one partition to operate on, so `=auto` (use job
  server / N processes) can't actually parallelise anything -- it's a
  no-op when paired with `=none`.
- Dropping `-fno-fat-lto-objects`. This controls intermediate `.o`
  *format* (slim bytecode vs fat bytecode-plus-machine-code) and is
  documented not to affect the linked binary when the linker plugin is
  loaded (`-fuse-linker-plugin` is, both before and after) -- the
  plugin consumes the bytecode and recompiles regardless.

So neither change *should* move binary size, and certainly not by 14%.
Two hypotheses, neither yet tested:

1. **Linker-plugin behaviour drifts with fat-vs-slim objects.** Possibly
   the plugin's symbol resolution pulls in more reachable code from
   the static archives (`libcrypto.a` is 28 MB, `libssl.a` 6 MB, etc.)
   when each `.o` carries its own machine-code copy alongside the
   bytecode. This would be a real behaviour gap, not a no-op.
2. **PGO profile variance**, downstream of test scheduling under `-m
   test --pgo`. Different profiles → different inlining → different
   code size. The
   [`2026-05-19T0223Z_x86_64.md`](./2026-05-19T0223Z_x86_64.md) report
   saw +0.9 MB on x86_64 from a much bigger recipe change (adding
   `-fuse-linker-plugin -fno-fat-lto-objects` outright), so +1.99 MB
   from removing one of those flags alone is on the high side for PGO
   noise but not absurd.

The +14.5% size jump came with **zero per-row CPU benefit** -- the
static absolutes are statistically the same as 2359Z. So whatever
caused the growth is dead weight at the moment.

### Concrete next step

Same-host A/B that AGENTS.md prescribes:

```sh
# revert configure-wrapper.sh CFLAGS/LDFLAGS to:
#   -flto -flto-partition=none -fuse-linker-plugin -fno-fat-lto-objects
# rebuild, capture size + numbers, compare.

docker compose exec -T spython sh -lc 'cd /workspace && \
  rm -rf deps-aarch64-linux-musl/Python-3.13.13 python-static-aarch64-linux-musl && \
  make USE_CROSSMAKE=1 python3 -j$(nproc)'
./benchmark/run.sh
size -A python-static-aarch64-linux-musl/bin/python3.13 | head -20
nm --size-sort python-static-aarch64-linux-musl/bin/python3.13 | tail -50
```

If the reverted binary is back to ~13.8 MB and per-row numbers match
2359Z, the new wrapper is regressing size for nothing and should be
reverted. If the reverted binary is also ~15.8 MB, the cause is PGO
profile drift and the new wrapper is exonerated.
