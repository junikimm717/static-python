## Analysis

**What this run is.** First PGO+LTO aarch64 report on a non-M4-Pro host
-- the prior canonical PGO+LTO baseline
(`2026-05-17T2146Z_aarch64.md`) is Apple M4 Pro via OrbStack, which
hides the PMU, cpuid, and cache sizes. This is Azure Neoverse-N1 (32
cores, bare-metal kernel, real `/proc/cpuinfo`, real PMU available
via `perf`), so it's the right place to (a) sanity-check whether the
M4 Pro PGO+LTO numbers generalise to other aarch64 implementations
and (b) finally have a host where the `perf stat` experiments the
previous reports queued up can actually run.

Two natural diff targets:

- **`2026-05-17T1907Z_aarch64.md`** -- same Azure Neoverse-N1 host,
  pre-PGO. Tells us how much the post-PGO+LTO recipe bought us here.
- **`2026-05-17T2146Z_aarch64.md`** -- M4 Pro, PGO+LTO. Tells us how
  much of the M4 Pro story is aarch64-general vs M4 Pro-specific.

### What PGO+LTO bought on this host (vs 1907Z, same Azure box)

Both interpreters were rebuilt. Static gained PGO (`--enable-optimizations`)
on top of its existing `-O3 -flto -ffunction-sections` recipe, plus
`-flto-partition=none`. Dynamic went from plain `-O3 -Wall` to upstream's
`--enable-shared --enable-optimizations --with-lto` (full PGO + `-flto
-ffat-lto-objects -flto-partition=none -fuse-linker-plugin` on libpython +
its bundled extensions). Absolute speedups since 1907Z:

The dynamic side got **more** uplift than the static side -- ~25% vs

linker-plugin LTO across libpython); static already had `-O3 -flto`
and gained only the PGO bit plus `-flto-partition=none`. So the
static-vs-dynamic *ratio* should narrow, and it does.

### `fib_recursive` regression replicated -- on a completely different aarch64 implementation

Most consequential single finding in this run.

| host                          | fib_recursive d/s |
|---                            |---:               |
| 1907Z Azure Neoverse-N1, **pre-PGO**        | 1.04x      |
| **2359Z Azure Neoverse-N1, PGO+LTO**         | **0.92x**  |
| 2146Z Apple M4 Pro, PGO+LTO                  | **0.78x**  |

Two independent aarch64 implementations (Neoverse-N1 in-order-issue
3.0 GHz cloud silicon vs Apple's M4 Pro Firestorm-class wide
out-of-order desktop silicon) both **got worse** at `fib_recursive`
when PGO+LTO was added. On the Azure box the regression is milder
(0.92x vs 0.78x) but it's clearly *the same effect*: same row, same
direction, same sign of change relative to the pre-PGO baseline.

That kills the "M4 Pro branch-target-buffer quirk" hypothesis the
2146Z report was hedging toward. The 2146Z report's other live
hypothesis -- "PGO+LTO over-inlines the recursive
`_PyEval_EvalFrameDefault` path, BTB mispredicts at varying recursion
depths" -- now applies to aarch64 broadly, not just one vendor's
microarchitecture. Concrete next step: this host has a real PMU
(unlike M4 Pro under OrbStack), so the `perf stat -e
branch-misses,iTLB-load-misses,L1-icache-load-misses` experiment the
last three reports queued up can finally run here, on both the static
and the dynamic binaries, against a tight `fib(20)` loop. That should
quantify the BTB story directly.

### `listcomp` headline win did NOT replicate

| host                          | listcomp d/s | static abs |
|---                            |---:          |---:        |
| 1907Z Azure Neoverse-N1, pre-PGO    | 1.14x | 64.9 ns |
| **2359Z Azure Neoverse-N1, PGO+LTO** | **1.07x** | **62.8 ns** |
| 2146Z Apple M4 Pro, PGO+LTO          | **1.27x** | 15.6 ns |

The 2146Z report's headline finding was a 1.03x -> 1.27x jump on
`listcomp` after enabling `-flto-partition=none` on static. On
Neoverse-N1 with the *same* recipe we get 1.07x -- nowhere near the
M4 Pro 1.27x. The static absolute on Azure barely moved (64.9 -> 62.8
ns), where on M4 Pro it dropped 18% (19.0 -> 15.6 ns). This is the
most interesting non-replication in the whole report.

Leading hypothesis: **L1i cache thrashing under aggressive
`-flto-partition=none` inlining**. M4 Pro's P-core L1i is 192 KiB;
Neoverse-N1's L1i is 64 KiB. `=none` keeps the whole program in one
LTO partition and inlines aggressively across procedure boundaries,
which bloats the hot `listcomp` body. On a 192 KiB L1i the inlined
version fits comfortably and the inlining pays for itself; on a 64
KiB L1i it falls out of cache and the extra fetch traffic eats the
inlining gains. The numbers are consistent with that: static `listcomp`
absolute is **4.0x slower on Azure than on M4 Pro** (62.8 ns vs 15.6
ns), and this is the *only* pure-libpython-internal row that didn't
benefit meaningfully from PGO+LTO on Azure.

Testable: `perf stat -e L1-icache-load-misses,L1-icache-loads` on
`listcomp` on both binaries on this host -- if static shows a
materially higher icache miss rate than dynamic, the L1i thrashing
story is confirmed. If miss rates are comparable, something else is
limiting the inlining payoff and we need to look at the disassembly
of the inlined hot loop.

### C-extension-boundary rows narrowed but still favor static

Three of the previous "consistent static win" rows from the README's
"What wins" table came in narrower on Azure than on M4 Pro:

| bench           | 2146Z M4 d/s | **2359Z Azure d/s** | what it stresses |
|---              |---:          |---:                 |---               |
| str_format      | 1.25x        | **1.11x**           | `_string.formatter_parser` C helper inlined into bytecode dispatch |
| json_roundtrip  | 1.22x        | **1.17x**           | `_json` C decoder cross-`.so` boundary |
| attr_access     | 1.17x        | **1.07x**           | `LOAD_ATTR` C specialisation path |
| except_path     | 1.19x        | **1.09x**           | `_PyErr_*` restore inlined across boundary |
| func_call       | 1.16x        | **1.06x**           | shallow Python-to-Python call dispatch |

Same pattern in each case: PGO+LTO closed the gap by giving dynamic
the same intra-libpython inlining quality. Static still wins because
it can LTO across the `_json.so`/`_string.so`/`_ssl.so` <-> libpython
boundary that the dynamic build cannot, but the margin is now in the
1.06-1.17x band on Neoverse-N1 vs 1.16-1.25x on M4 Pro. The
implication: roughly half of the M4 Pro "C-extension-boundary" win was
microarchitecture-amplified, not structural.

Same flavor of narrowing as the CPU side. Static still wins
consistently on `stdlib` (1.13x) -- frozen modules + no `dlopen` is
intrinsic to the link mode and PGO can't take it away. The `bare`
and `sysmod` rows narrowed more because the static gain there was
mostly about not paying per-`.so` mmap/init, and the dynamic side
just got faster at the per-`.so` init work via PGO training on the
import path.

### `system / static = 1.41x`

Same caveat as 2146Z (and 2112Z): Alpine apk's `/usr/bin/python3` is
unchanged -- still `-Os`, no PGO, 3.12.13 -- so as we get faster the
ratio widens automatically. `str_format` vs system is now **2.30x**,
`except_path` **2.11x**, `json_roundtrip` **1.75x**, `regex_match`
**1.66x**. Still useful as a sanity check that the static binary
isn't catastrophically wrong on any single row; not useful as a
meaningful comparator. The README correctly downgraded
`system/static` to a non-substantive ratio in the 2146Z update.

### What's missing now

The PMU-on-real-hardware availability this host gives us reshuffles
priorities relative to the README's current list:

1. **`perf stat -e branch-misses,iTLB-load-misses,L1-icache-load-misses`
   on `fib_recursive`** on both static and dynamic on this host. With
   two independent aarch64 implementations now showing the same
   PGO+LTO-introduced regression, this is the experiment that
   identifies the actual mechanism. Promoted above DYNAMIC_USE_INTREE.
2. **`perf stat -e L1-icache-load-misses,L1-icache-loads` on
   `listcomp`** on this host. The L1i-thrashing hypothesis is the
   simplest explanation for the M4-Pro-vs-Neoverse-N1 non-replication;
   one `perf` invocation confirms or falsifies it.
3. **DYNAMIC_USE_INTREE=1 dynamic build** -- still relevant, same
   rationale as the README. Now even more interesting because the new
   geomean is small enough (1.07x on this host) that isolating
   asymmetries 2+3 vs asymmetry 1 may show that asymmetry 1 alone is
   ~zero on aarch64.
4. **STATIC_NO_LTO=1 static build** -- mirror, same as README.
5. Cross-arch PGO via qemu-user; same as README.
