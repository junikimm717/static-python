## Analysis

Re-run on the same Azure Xeon Platinum 8370C (Cascade Lake, `1013-azure`
kernel) as the deleted 2322Z report. No intentional build change; binaries
are identical to 2322Z (static still 16.0 M; dynamic baseline still 30.9 M).
The primary purpose is reproducibility confirmation and locking in a stable
Cascade Lake baseline now that 2322Z is gone.

### CPU geomean reproduces at 1.20x

The headline number matches 2322Z exactly. That puts the four PGO+LTO data
points at N1 1.07x / M4 Pro 1.11x / Zen 5 1.18x / Cascade Lake 1.20x --
unchanged from the cliffs-notes range.

### Per-row moves: mostly VM noise, two rows worth flagging

Azure shared VMs carry ±5-10% run-to-run variance on individual rows even
when the geomean is stable. Most of the per-row moves here fall inside that
band and should not be used as priors for mechanism hypotheses. Two rows are
at the edge of that window:

| bench | 2322Z (deleted) | 0223Z (this) | delta |
|---|---:|---:|---:|
| `regex_match` | 1.12x | **1.22x** | +10 pp |
| `arith_loop` | 1.47x | **1.38x** | −9 pp |
| `func_call` | 1.25x | 1.32x | +7 pp |
| `fib_iter` | 1.27x | 1.22x | −5 pp |
| everything else | — | — | ≤4 pp |

`regex_match` at 1.22x is notably higher than the prior Cascade Lake value
and now matches Zen 5 and the two aarch64 numbers (1.07-1.22x). The 1.12x
in 2322Z would have been the low end of the cross-host spread for that row;
the 1.22x here is consistent with the rest of the cohort. It is more likely
that 2322Z was a low outlier on this row than that the static `_sre` path
regressed and then recovered. Worth a dedicated stability run if the row
matters for a hypothesis.

`arith_loop` at 1.38x is the lowest Cascade Lake value for that row across
all runs. The 1.47x in 2322Z was itself an outlier on the high end vs Zen 5
1.52x and aarch64 1.10-1.14x; 1.38x is closer to a neutral landing for this
µarch. Same noise caveat applies.

### `fib_recursive` deepened: 0.89x → 0.85x

The cross-host regression table now reads 0.78x (M4 Pro) / 0.85x (Cas) /
0.89x (Zen 5) / 0.92x (N1). The Cascade Lake value moved in the wrong
direction since 2322Z by 4 ratio-percent. Given the VM noise level, a 4 pp
move on a single row is borderline, but the direction is consistent with
every other host showing the same loss. Nothing about the binary changed
between 2322Z and 0223Z, so this is noise, not a new signal. The promoted
next step from 0000Z remains live: `perf stat -e cycles,instructions,
branch-misses,iTLB-load-misses` on `fib_recursive` on Cascade Lake or
Zen 5.

### `stdlib` startup 1.14x confirms Zen 5 0000Z was the outlier

| scenario | 0223Z | 2322Z (deleted) | 0000Z Zen 5 |
|---|---:|---:|---:|
| bare | 1.07x | ~1.07x | 1.06x |
| sysmod | 1.08x | ~1.08x | 1.08x |
| stdlib | **1.14x** | 1.14x | **0.89x** |
| geomean | **1.10x** | ~1.10x | 1.01x |

Cascade Lake and Zen 5 have the same 32 KiB L1i, so cache topology alone
cannot explain the Zen 5 anomaly. The 1.14x value has now been confirmed on
Cascade Lake twice, sits inside the M4 Pro (1.13x) and N1 (1.13x) band, and
makes the Zen 5 0.89x look increasingly like a one-time scheduling artefact
on a busy local machine rather than an `stdlib`-import behaviour difference.
The `strace -c` triage item in the cliffs-notes open experiments list can be
downgraded from "one-hour answer" to "low priority unless the 0000Z Zen 5
run repeats it".

### Next steps (unchanged from 2322Z)

1. `perf stat` on `fib_recursive` on a real-PMU x86_64 host (Cascade Lake
   or Zen 5). The four-host consistent loss and the N1 branch-mispredict
   finding from the [0024Z perf run](./2026-05-18T0024Z_aarch64_perf.md)
   both point at `_PyEval_EvalFrameDefault` dispatch; an x86_64 PMU can
   distinguish BTB-vs-RAS-vs-conditional that the N1 measurement (which
   fences indirect branch counters on Azure) could not.
2. `DYNAMIC_USE_INTREE=1` -- the widest surviving wins (`arith_loop`,
   `func_call`, `json_roundtrip`, `str_format`) are all C-extension-
   boundary rows; linking the dynamic baseline against LTO'd in-tree libs
   isolates how much of those wins come from whole-program scope vs
   Alpine's libs being un-LTO'd.
3. `STATIC_NO_LINKER_PLUGIN=1` same-host A/B. The 2322Z-to-0223Z delta
   is indistinguishable from noise, so the plugin contribution has still
   never been isolated cleanly.
4. Same-host stability run: rebench in a separate window on the same Azure
   host to band the per-row noise estimates before committing to any
   `regex_match` or `arith_loop` hypothesis.
