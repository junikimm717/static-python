## Analysis

Fresh AWS aarch64 run on an ARM Neoverse-V2 host after rebuilding the dynamic
baseline on this machine. This replaces the discarded 14:14Z report from the
same host: that first run used the same code but the static binary still had
DWARF debug sections. The unstripped artifact was backed up as
`python-static-aarch64-linux-musl/bin/python3.13.unstripped-20260523T141849Z`,
then the live `python3.13` was stripped and rebenchmarked.

The size row is now clean again: static is **15.8M**, matching the latest
Neoverse-N1 aarch64 report and the current x86_64 static size class. That
falsifies the scary 60.9M size from the discarded run as debug-symbol bloat, not
a real link-size regression.

CPU geomean lands at **1.07x dynamic/static**, effectively the same as the
Neoverse-N1 server-ARM band (1.07-1.09x) and below M4 Pro (1.11x). Absolute
times are much better than N1, as expected for Neoverse-V2, but the
static-vs-dynamic shape is familiar: every CPU row wins except `fib_recursive`,
which is still negative at **0.90x**. That keeps the existing `fib_recursive`
diagnosis alive: it is a stable recipe/pathology, not a one-host anomaly.

Two CPU rows move README bounds. `json_roundtrip` is **1.11x**, below the
previous PGO+LTO lower bound of 1.17x, so JSON is not always a large
whole-program-LTO win. `attr_access` is **1.04x**, also a new low for the
post-PGO+LTO set. The overall aarch64 story stays conservative: static is a
repeatable high-single-digit CPU win on server ARM, not a 20%+ win.

Startup is the one surprising part: **1.17x** geomean, with `stdlib` at
**1.24x**. The stripped binary is slightly slower in absolute startup than the
discarded unstripped run, but dynamic's minimum samples moved more, especially
for `stdlib`. Since stripping should not make dynamic slower, treat the exact
startup delta as run-to-run noise or host scheduling until repeated. Direction
still supports the broad pattern: Zen 5 remains the only startup-loss outlier.
