## Analysis (LLM generated)

First clean post-toolchain-bump run on this host. Static = gcc 15.1.0
+ `-O3 -flto`; dynamic = gcc 15.2.0 + Python autoconf defaults (no
`--enable-optimizations`, no `--with-lto`); system = Alpine apk 3.12.13
built with `-Os -O2 -fno-plt`. None of the three uses `-march=native`,
so static and dynamic both emit generic x86_64-v1 codegen and the
host µarch only shows up in execution.

The only meaningful in-tree comparator on disk is the
**1818Z** report from this morning -- same source, same toolchain, but
running on an Azure Xeon Platinum 8370C (Cascade Lake, kernel 6.17)
instead of this Ryzen AI 9 HX 370 (Zen 5, kernel 6.18). So absolute
ns/op deltas vs 1818Z are mostly host change; ratio deltas isolate
what's actually driven by linkage.

### CPU geomean firmed up vs same-version dynamic

dynamic/static: 1818Z **1.17x -> 1.18x**, system/static: 1818Z **1.14x
-> 1.25x**. The same set of rows wins by similar margins across the
host change, which is what we want: the LTO-buys-you-inlining story is
not Cascade-Lake-specific.

| bench | 1818Z (Xeon) | 1924Z (Ryzen) |
|---|---:|---:|
| except_path  | 1.43x | **1.45x** |
| arith_loop   | 1.40x | **1.45x** |
| func_call    | 1.42x | **1.38x** |
| fib_iter     | 1.26x | **1.28x** |
| listcomp     | 1.18x | **1.26x** |
| dictops      | 1.28x | **1.21x** |
| attr_access  | 1.20x | **1.23x** |
| regex_match  | 1.14x | **1.21x** |

These are the eval-loop and dispatch-heavy rows. They are the *real*
signal of "what does statically linking + LTO buy you?" -- a ~20-45%
margin over the same source built dynamically with the same toolchain,
on either µarch.

The two rows that closed to parity (`fib_recursive` 0.99x -> 0.98x,
`str_format` 0.99x -> 0.97x) are uninteresting: they were at parity
before the host change and they're at parity after. Whatever
characteristic of the static build helped on the other rows just
doesn't reach the C-level recursion path or the `PyUnicode_Format`
dispatch.

### json_roundtrip is the one real regression, and it is robust

| | 1818Z (Xeon) | 1924Z (Ryzen) |
|---|---:|---:|
| static absolute   | 13.11 us | **8.94 us** |
| dynamic absolute  | 10.09 us | **6.70 us** |
| dynamic/static    | 0.77x    | **0.75x** |

Same magnitude on two different µarches and against two different
gcc point releases (15.1.0 static / 15.2.0 dynamic both runs). The
hypothesis from the previous report -- "regressions in C-extension
rows are about library-version skew" -- was already falsified once
we matched openssl/sqlite/libffi/xz; now it's also falsified for the
host. What is left is "static + `-O3 -flto` on the JSON decoder path
takes ~25-30% longer than the dynamic build's per-TU compilation
does, on any x86_64 host we've measured." That is the next thing to
actually instrument.

### Startup wins are real but smaller

dynamic/static geomean **1.13x -> 1.17x**, system/static **1.02x ->
1.07x**. The static binary leads on every scenario (bare 1.12x,
sysmod 1.22x, stdlib 1.18x). Absolute static numbers are all faster
than the Xeon report (bare 11.49 -> 10.25 ms, stdlib 20.80 -> 17.69
ms) -- partly Zen 5, partly NVMe vs Azure VHD, partly the fact that
mmap-and-go is harder to slow down than it is to speed up.

The system python (Alpine apk) is now within 7% of our static binary
on startup geomean, which makes its position as the "external
baseline" weak: at startup it's losing to us by less than the
3rd-decimal-place noise on a 40-sample min, and at CPU it's losing
on a fight where it's been deliberately built `-Os` and we've been
built `-O3 -flto`. See below.

### What "system python" is and isn't

For anyone reading the system/static column and assuming we're
beating an optimized stock interpreter: we aren't. Alpine's
`/usr/bin/python3` was built with

    PY_CFLAGS_NODIST: -Os -fstack-clash-protection -Wformat
                      -Werror=format-security -fno-plt -g -O2 ...
    LTO             : (none)
    OPT             : (no --enable-optimizations)

i.e. size-optimized for the apk packaging policy, no PGO, no LTO.
That makes the 1.25x geomean half-real-win, half-rigged-baseline.
When system *beats* us on a row (`fib_recursive` 0.95x,
`json_roundtrip` 0.82x), it does so *despite* being detuned, which
makes those regressions look slightly worse for us than they would
look against a proper baseline.

The right next comparator is a same-source dynamic CPython built
with `--enable-optimizations --with-lto`. That probably halves our
visible "static vs dynamic" margin on the eval-loop rows, because
PGO genuinely accelerates the bytecode dispatch loop -- and that's
the thing whose margin we are currently claiming.

### Next probes

1. **PGO dynamic baseline.** Add a fourth interpreter:
   `python-dynamic-pgo-${HOST_ARCH}-linux-musl/bin/python3.13`,
   configured `--enable-shared --enable-optimizations --with-lto
   --without-ensurepip`. Bench it as a fourth column. That replaces
   the apk "rigged" baseline with an apples-to-apples one.
2. **`perf stat` on `json_roundtrip`.** Run
   `perf stat -e cycles,instructions,branch-misses,L1-icache-load-misses,iTLB-load-misses`
   on `python -c 'import json; data=open("big.json").read(); [json.loads(data) for _ in range(N)]'`
   against the static and dynamic binaries. Settles whether static
   retires *more instructions* (LTO making the decoder bigger) or
   *the same instructions slower* (icache pressure).
3. **Repeat-build variance characterization.** Run 3-5
   `make clean && make python3 && ./benchmark/dynamic-build.sh && ./benchmark/run.sh`
   cycles and report the median-of-medians per row. CPython
   microbenchmarks against statically linked binaries with
   `-ffunction-sections` + `--gc-sections` are known to be sensitive
   to relink layout on the tight recursive rows; the report should
   reflect that band rather than a single build's best-of-7.
