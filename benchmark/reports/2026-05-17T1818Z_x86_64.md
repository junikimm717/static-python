## Analysis

First post-bump run after **gcc 9.4 -> 15.1.0** (musl-cross-make pin in
`cross-make/config.mak`) and the library bumps **openssl 3.5.0 -> 3.5.6,
sqlite 3.49.2 -> 3.51.2, libffi 3.4.8 -> 3.5.2, xz 5.8.1 -> 5.8.3** -- all
to match Alpine's dynamic stack so the comparison isolates link mode rather
than vintage. Compiler versions are now within a point release of each other
(static gcc 15.1.0 vs dynamic gcc 15.2.0); the libraries are upstream-matched
to the apk-installed `*-dev` packages our `dynamic-build.sh` links against.

What moved relative to the previous (gcc-9.4) baseline:

- CPU geomean **1.09x -> 1.17x** vs same-version dynamic. The eval-loop
  benches now turn in the kind of margins LTO is supposed to buy you:
  `except_path` 1.43x, `func_call` 1.42x, `arith_loop` 1.40x, `dictops`
  1.28x, `fib_iter` 1.26x.
- `str_format` (0.95x -> 0.99x) and `fib_recursive` (0.92x -> 0.99x)
  effectively closed -- those *were* compiler-version artefacts on the
  static side, and they vanish once both sides use gcc 15.
- `json_roundtrip` is the lone holdout: still ~0.77x vs dynamic, with
  versions matched. That falsifies the previous README hypothesis that the
  C-extension regressions were about library versions. Whatever this is, it
  is intrinsic to static + `-O3 -flto` on the JSON decoder path.
- Startup geomean **1.10x -> 1.13x** vs dynamic, ~1.02x vs system. As
  expected from frozen stdlib + no `dlopen`; the gcc bump moves it less than
  it moves the CPU side.

Things this run does *not* explain and that are worth poking next time:

- `json_roundtrip` is reproducibly ~25% slower static. Worth profiling: is
  it an icache miss story (15 MB LTO'd binary vs Python's modest hot
  set), bad code-layout inlining around the recursive decoder, or
  something concrete like `PyUnicode_FromKindAndData` getting a worse
  specialisation under `-fvisibility=hidden`? `perf stat -e
  cache-misses,instructions` on the two binaries running
  `json.loads(big_doc) * N` should settle it.
- The system python (3.12.13) tracks the dynamic 3.13.13 closely on most
  rows, which suggests Alpine is *not* shipping a PGO build (or if it is,
  the PGO profile isn't representative). Worth confirming -- `python3 -c
  'import sysconfig; print(sysconfig.get_config_var("PY_CORE_CFLAGS"))'`
  on `/usr/bin/python3` would settle it.
- Cross-arch is still on the old toolchain; once we rebuild the other 10
  archs under gcc 15 + the bumped libs, we should re-bench at least one
  aarch64 host so the README claim that "this is the win on x86_64" gets a
  second data point.
