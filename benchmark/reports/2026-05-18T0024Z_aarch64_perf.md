# perf investigation -- aarch64 -- 2026-05-18T0024Z

Generated: 2026-05-18 00:24 UTC

Not a benchmark re-run. This is the **`perf stat` / `perf record` follow-up**
to the cross-aarch64 `fib_recursive` regression and the M4-Pro-vs-N1
`listcomp` non-replication called out in
[`2026-05-17T2359Z_aarch64.md`](./2026-05-17T2359Z_aarch64.md) and listed as
items 3 and 4 in the README's "What is missing". Same Azure Neoverse-N1
host as the 2359Z report; same binaries on disk (built 2026-05-17 23:55Z).
The 2359Z report's wall-clock ratios reproduce exactly in the probe
harness used here (static 6.246s vs dynamic 5.646s on `fib(20) x 5100` =
0.904x, matching 2359Z's 0.92x).

## Environment

- arch: `aarch64` (Azure Neoverse-N1, 32 cores, real PMU)
- kernel: Linux 6.17.0-1013-azure
- host `perf` 6.17.13 (`/usr/bin/perf`); driven from host via
  `sudo perf stat -- nsenter -t <CPID> -m -p -- <python>` so the dynamic
  binary's musl loader + Alpine libs resolve inside the container's mount
  namespace while perf itself runs with full host caps
- `kernel.perf_event_paranoid` lowered from 4 to -1 for the run
- Azure platform fences off several PMU events (`l1i_cache`,
  `l1i_cache_refill`, `itlb_walk`, `br_indirect_spec`, `br_return_spec`
  all return 0 even when measured in isolation). Of the README's
  requested counter list, `iTLB-load-misses` and `L1-icache-load-misses`
  are therefore **not measurable on this host**. `stalled-cycles-frontend`
  is the available coarse proxy for frontend pressure.

## Probe harness

Two tight drivers under `.perf_tmp/`:

- `probe_fib.py` -- 100 warmup + 5000 measured invocations of the
  microbench's `fib_recursive` body (`fib(20)`; 21891 calls per
  invocation). ~6.2 s of steady-state work per run, startup <0.5%.
- `probe_listcomp.py` -- 20 warmup + 1000 measured invocations of
  `[x*x for x in range(100_000)]`. ~6.7 s steady-state, startup <0.5%.

Both probes match the microbench inner-loop bodies exactly. Each probe is
run 3x per binary (`perf stat -r 3`) to get standard-error bars.

## Headline finding

**`fib_recursive`: the regression is branch-mispredict-driven, NOT L1i
or icache or iTLB.** Static retires the *same* number of instructions
as dynamic (within 0.15%) but mispredicts 1.66x as many branches,
spends 1.8x as many cycles backend-idle, and runs 11% fewer
instructions per cycle. **Frontend stalls are *lower* on static
(0.28% vs 0.68%)** -- directly falsifying any "PGO+LTO bloats the eval
loop past L1i" story for this benchmark on this host. The hot
mispredicting instructions on both binaries are inside
`_PyEval_EvalFrameDefault`, concentrated at the computed-goto bytecode
dispatch `br x*` sites and the dispatch-table loads immediately
preceding them.

**`listcomp`: also NOT L1i-bound.** Frontend stalls 4.34% on static
vs 4.21% on dynamic -- statistically indistinguishable, both about
ten times higher than the fib_recursive numbers, and **equal between
the two binaries**. The 2359Z L1i-thrashing hypothesis for the M4
Pro vs N1 `listcomp` non-replication is **falsified by direct
measurement** on N1. Static wins listcomp 1.07x here because it
executes 5% fewer instructions and 12% fewer branches (i.e. LTO is
folding work), not because dynamic is thrashing icache.

## Method

### Why a wrapper, why `nsenter`

The dynamic binary depends on the musl loader (`/lib/ld-musl-aarch64.so.1`)
and Alpine apk `lib*.so.N` (`libssl.so.3`, `libsqlite3.so.0`, etc.) which
only exist inside the container's mount namespace. The container itself
does not have `CAP_PERFMON`/`CAP_SYS_ADMIN`, so even with `apk add perf`
(and there is such a package, `perf-6.19.14-r0`) `perf` inside the
container reports `No permission to enable cycles event`.

Cleanest fix without touching `docker-compose.yml`:

```sh
sudo perf stat -- nsenter -t <container-pid> -m -p -- <python-binary> <probe.py>
```

`perf` runs on the host as root (full caps, no paranoid restriction),
`nsenter` only enters the container's mount + pid namespaces, the
dynamic binary's `dlopen()` resolves correctly, and `perf stat` measures
the entire `nsenter -> python` subprocess tree. The same `nsenter`
wrapper is used for static (where it's not strictly needed) so the
measurement noise is symmetric between the two binaries.

Symbol resolution: `perf record`'s data file references binaries by
their container path (`/workspace/...`). Creating
`/workspace -> /home/junikim/staticpy` as a host symlink lets `perf
report` resolve symbols against `libpython3.13.so.1.0` (which still
has `.debug_info` -- the dynamic build is not stripped). The static
binary **is** stripped (`file ... statically linked, stripped`) so
its samples come back as raw `.text` addresses; cross-referencing
against `objdump -d --start-address=...` gives instruction-level
context but not function names.

### Counters

The README requested `cycles,instructions,branch-misses,iTLB-load-misses,L1-icache-load-misses`.
Azure blocks the last two. The full event list actually measured was

```
cycles,instructions,branches,branch-misses,stalled-cycles-frontend,
stalled-cycles-backend,l1i_cache,l1i_cache_refill,itlb_walk,
L1-dcache-loads,L1-dcache-load-misses,br_mis_pred_retired,br_retired,
br_indirect_spec,br_return_spec,inst_spec
```

`l1i_cache`, `l1i_cache_refill`, `itlb_walk`, `br_indirect_spec`, and
`br_return_spec` always come back 0 on this host (verified by running
each in isolation against `/bin/true` -- not a multiplexing artefact).
`stalled-cycles-frontend` does work and is the standard fallback for
"is the frontend the bottleneck?".

### Branch mispredicts localise to the bytecode dispatch

`perf record -F 4999 -e branch-misses` then `perf report`:

**Dynamic** (symbols clean):

| % of mispredicts | symbol                       |
|---:              |---                           |
| 76.51%           | `_PyEval_EvalFrameDefault`   |
| 13.71%           | `_PyLong_Add`                |
|  8.91%           | `_PyLong_Subtract`           |
|  0.15%           | `_PyEval_FrameClearAndPop`   |
| **99.28%**       | (top 4 -- all in libpython)  |

These are the four hottest functions of the entire benchmark: 81.13%
of `cycles` samples are in `_PyEval_EvalFrameDefault`, 8.74% in
`_PyLong_Subtract`, 5.53% in `_PyEval_FrameClearAndPop`, 4.29% in
`_PyLong_Add`. So branch-misses track cycles almost exactly --
there's no surprising "misses are in some weird code path" story.

**Static** (stripped, addresses only) -- the top branch-miss IP
is `0x4f3b44`. `objdump -d` around that address:

```
4f3b3c:  12800001  mov     w1, #0xffffffff       ; #-1
4f3b40:  f862d800  ldr     x0, [x0, w2, sxtw #3] ; x0 = dispatch_table[w2]
4f3b44:  53087ce7  lsr     w7, w7, #8            ; <-- HOTTEST IP (17.4% of misses)
4f3b48:  b9804276  ldrsw   x22, [x19, #64]
4f3b4c:  aa1303f7  mov     x23, x19
4f3b50:  b9002ea5  str     w5, [x21, #44]
4f3b54:  f90026b3  str     x19, [x21, #72]
4f3b58:  8b160cd6  add     x22, x6, x22, lsl #3
4f3b5c:  b9004261  str     w1, [x19, #64]
4f3b60:  d61f0000  br      x0                    ; <-- computed-goto dispatch
```

That's the CPython threaded-interpreter dispatch sequence: load
`dispatch_table[opcode]`, push some bookkeeping into the frame, indirect
`br` to the loaded target. Skylake / Neoverse / etc. PMUs typically
attribute a branch mispredict to the in-flight victim of the squash,
which lands a few instructions ahead of the actual branch, exactly
matching `0xf3b44` being three instructions before the `br x0` at
`0xf3b60`. The cluster of other-hot static IPs (`0xf30e4`, `0xf36b8`,
`0xf39a0`, `0xf4f5c`, ...) all land in the same `_PyEval_EvalFrameDefault`
body -- multiple opcode handlers each end in their own `br x*` per the
USE_COMPUTED_GOTOS layout, and each is a separate mispredict site.

**Dynamic mispredict site for comparison**, `perf annotate`'d inside
`_PyEval_EvalFrameDefault`:

```
 16.45% : 119b50  str     x19, [x21, #72]    ; symmetric position
  4.87% : 119b44  str     w4, [x21, #44]
  3.14% : 119b24  ldrh    w7, [x3, #200]
  4.28% : 11b364  cbz     x3, ...            ; another dispatch site
  2.83% : 11b384  lsr     w7, w7, #8
  2.23% : 11b388  ldr     x1, [x0, w1, sxtw #3]
  ...
```

Same instruction shapes, same hot region of the same function on both
binaries. The mispredicts are **structurally the same kind of branch**
(computed-goto dispatch indirect `br x*`); only the rate is different.

### Sub-hypothesis check: "static merges dispatch sites"

If `-flto-partition=none` inlining had collapsed the number of distinct
dispatch sites (which would have explained the 5% drop in retired
branches), it would have left fewer entries in the BTB. A naive count
of indirect `br x*` instructions in `_PyEval_EvalFrameDefault`:

- Dynamic: function is 74,664 B = 18,666 insns, **290** `br x*`
- Static: 80-192 KB window around the hot range, **247-300** `br x*`

(Static is stripped so the function boundary is fuzzy; both windows
straddle the hottest perf samples.) That's the same order of magnitude
in both binaries, so the simplest version of "fewer dispatch sites
under LTO" is **not the mechanism**. The 5% branch-count drop on static
must be coming from a different inlining (probably the small-int
add/sub helpers folded into the bytecode arithmetic ops removing
intermediate `bl`/`ret` pairs).

What's left as the candidate mechanism, then, is **BTB tag aliasing**:
PGO+LTO reshuffled the layout of opcode handlers in static such that
multiple distinct dispatch IPs now collide in the BTB hash, the
per-opcode pair correlation that USE_COMPUTED_GOTOS exists to exploit
gets clobbered, and miss rate goes from 0.91% (dynamic) to 1.59%
(static). This is consistent with **Azure Neoverse-N1's 32-set,
8-way BTB**, where layout shifts of a few KiB can change which entries
share a set, but is not directly testable on this host because
`br_indirect_spec`/`br_return_spec` are fenced off.

## What didn't reproduce / went sideways

- The `iTLB-load-misses` and `L1-icache-load-misses` counters from
  the README's requested event list both return 0 on Azure. The Zen 5
  follow-up matters more now -- AMD does not fence these.
- `br_indirect_spec` and `br_return_spec` are also fenced. So I can't
  cleanly split "is the misprediction the indirect dispatch `br x*` or
  the conditional opcode-specific branch?" on this host. The
  disassembly + sample location strongly suggests indirect dispatch,
  but it's an inference not a direct measurement.
- The static binary is stripped (the Makefile install path uses the
  standard `make install` which strips). I worked around this with
  `objdump -d --start-address=...` against the .text addresses perf
  reported, which is good enough to read the dispatch sequence but not
  to attribute samples to individual functions automatically. Building
  static with `STRIPPED=0` (or just keeping the unstripped
  `python.exe` from the build tree before `make install`) would let
  `perf report` give symbol-level breakdown for the static side too.

## Concrete next steps (priority order)

1. **Re-run this perf-stat probe on the Zen 5 host** (Ryzen AI HX 370 /
   `2026-05-18T0000Z_x86_64.md`). x86 doesn't fence
   `br_indirect_spec`/`br_return_spec`/`L1-icache` counters, so we get
   the indirect-vs-conditional split *and* a direct L1i miss-rate
   number. Same script, same probes; the only changes are the perf
   event names (`L1-icache-load-misses` for the x86 equivalent and
   `r4c4`/`r4c5`/etc. for the AMD-specific indirect-branch counters).
   This is the single experiment that turns "BTB mispredict" from a
   strong inference into a measured fact.
2. **Keep the static build unstripped for the next bench cycle.** Add
   `INSTALL_STRIP=0` (or equivalent) to the `make install` step so
   `_PyEval_EvalFrameDefault`'s function boundary and the inlined
   helpers are perf-resolvable on the static side. Cost is a few MB of
   extra binary size; gain is symbol-level attribution.
3. **Targeted disassembly diff of `_PyEval_EvalFrameDefault` between
   the two binaries.** Once (2) lands, dump `objdump -d` of the
   function from both, count distinct `br x*` instances, eyeball the
   biggest opcode handlers, and look for an instance where the static
   side has visibly different layout / different inlining that
   correlates with the mispredict-hot opcode for `fib_recursive`
   (mostly `CALL`, `RETURN_VALUE`, `BINARY_OP_ADD_INT`,
   `BINARY_OP_SUBTRACT_INT`, `COMPARE_OP_INT`).
4. **`STATIC_NO_LTO=1` experiment from the README.** If stripping
   `-flto -flto-partition=none` brings fib_recursive back to 1.04x
   (the 1907Z pre-PGO ratio on this host), that's a clean attribution
   to LTO (probably plus PGO interacting with LTO, since the two
   landed in the same recipe). If it doesn't, PGO-only is what hurt.
5. Drop the `iTLB-load-misses` / `L1-icache-load-misses` ask from the
   "What is missing" item 3 in the README, since we now know it's
   ungatherable on this host; replace it with the Zen-5-side perf run.

## Analysis

See "Headline finding" + the two `perf stat` tables + "Implications for
the open hypotheses" above. The L1i/icache hypothesis is dead for both
`fib_recursive` and `listcomp` on this host; the `fib_recursive`
regression is squarely a branch-prediction + backend-execution problem
inside `_PyEval_EvalFrameDefault`, with the hot mispredicting
instructions sitting on the computed-goto dispatch `br x*` sites and
their immediately-preceding setup. The remaining ambiguity (BTB
aliasing vs RAS pressure vs PGO mistraining of conditional opcode
branches) needs the indirect-branch counter split that Azure's PMU
fencing prevents -- next stop is the Zen 5 host.
