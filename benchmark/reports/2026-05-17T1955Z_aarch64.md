-- no Makefile, toolchain, or `python/` edits since; `git status` only shows
`README.md` modified. The diff that matters is the **host**: previous run
was an Azure Neoverse-N1 VM (32 cores, 6.17 kernel); this one is an Apple
**M4 Pro** under OrbStack's Linux VM (14 logical cores, 6.19 kernel). So
absolute ns/op deltas vs the prior aarch64 report are silicon noise rather
than a code regression and should not be read as a re-run -- the table
below is a fresh single-machine measurement.

Environment-detection observations from this machine:

- `cpu_model()`'s Apple-silicon branch matches implementer `0x61` but only
  knows parts `0x022/0x028/0x029/0x032` (M1 family + M2). OrbStack
  publishes `CPU implementer: 0x61 / CPU part: 0x000` in the guest's
  `/proc/cpuinfo` -- the part ID isn't a new Apple core we forgot to map,
  it's *hidden by the hypervisor*. Adding entries for M3/M4/M4-Pro parts
  to the case statement would not help; the fix is either a special-case
  for `0x61/0x000` ("Apple silicon, part hidden by hypervisor") or
  shelling out to a host-side helper. Patched the cpu line by hand here.
- `/sys/devices/system/cpu/cpu0/cache/index*/` exists but has only
  `level`, `type`, and `shared_cpu_list` -- no `size` file. So
  `cache_size_at()` correctly returned `?` for every level. This is
  OrbStack's virtual sysfs, not a script bug; on the Neoverse-N1 host
  the sizes populated as expected. The pragmatic fix is to additionally
  probe `lscpu -B --caches` (which falls back to cpuid on x86 and to
  CCSIDR_EL1 on aarch64 where the kernel allows it) and accept a `?`
  when both paths fail. Patched the cache line by hand here using host
  `sysctl hw.perflevel*` values; only the P-core line is what the static
  build will run on under any meaningful load, since OrbStack pins to
  P-cores for foreground work.

What this single-machine static-vs-dynamic comparison says (both 3.13.13,
both built against the same OpenSSL 3.5.6 / SQLite 3.51.2 / readline 8.3 /
zlib 1.3.2 pins, so link mode is the only free variable between `static`
and `dynamic`):

- **Static is 1.24x faster geomean, slightly wider than Neoverse-N1's
  1.16x.** Same shape as before: the bytecode-eval rows (`fib_iter`,
  `arith_loop`, `listcomp`, `dictops`, `attr_access`) cluster at
  1.13x-1.33x, and the same C-helper-heavy rows take the headline wins --
  `str_format` **1.71x** (vs 1.31x on Neoverse-N1), `except_path`
  **1.69x** (vs 1.24x), `func_call` **1.41x** (vs 1.10x), `json_roundtrip`
  **1.27x** (vs 1.30x). The pattern is consistent with cross-module
  inlining: hot paths that traverse the bytecode-dispatch -> C-helper
  boundary benefit most on a CPU with a deeper reorder buffer and more
  aggressive branch predictor (M4 Pro), where the inlined-and-LTO'd
  static build can keep more dependent ops in flight.
- **The one row where static *loses* hard: `fib_recursive` 0.69x** (28.2
  ns/call static vs 19.4 ns/call dynamic vs 19.2 ns/call system). The
  Neoverse-N1 run had this row essentially flat at 1.04x, so this is a
  silicon-specific inversion, not a regression we introduced. The
  hypothesis worth testing: deep-recursion Python frame allocation
  (`_PyEvalFramePushAndInit` -> `_PyFrame_PushUnchecked` ->
  `_PyThreadState_PushFrame`) is exactly the chain LTO most aggressively
  inlines into the recursive call site in the static build. That bloats
  the code at the call site; the M4 Pro's larger L1i (192K P-core) is
  plenty to hold it, so it isn't an icache miss -- but the M4's
  branch-target predictor punishes mixed return/call patterns at varying
  recursion depths more than the Neoverse-N1's does. The same row's
  `system` python (Alpine 3.12.13) lands at 19.2 ns, equally fast as
  `dynamic`, which makes "static-build code bloat at the recursive call
  site" the more parsimonious explanation than "M4 Pro is just bad at
  function calls" -- because `func_call` on the *same* hardware is
  46.6 ns static / 65.9 ns dynamic, i.e. static wins by 41% on shallow
  calls. Shallow vs recursive is the discriminator.
- **Startup: system python wins on bare/sysmod (0.90x, 0.91x), unlike
  Neoverse-N1 where it was within 1%.** Absolute startup is much lower
  here (5.4 ms vs 13.2 ms bare), which amplifies any constant-cost
  difference -- e.g. musl's dynamic loader resolving `libpython.so` plus
  its 6 MB of relocations is much cheaper relative to the 5 ms total than
  it was relative to 13 ms. The static build is still ahead of *dynamic*
  on all three scenarios (1.16x, 1.18x, 1.33x), so the "no-`dlopen` is
  faster on cold start" story is intact; the system python's edge is
  about it being a 5.9 MB libpython rather than our 31 MB libpython, so
  fewer relocations to chase. Worth a quick `LD_DEBUG=statistics`
  comparison to confirm.
