## Analysis

First reproducible **gcc 15.1.0** build on aarch64 in this tree.
Readline 8.2 doesn't compile under gcc 15's C23 default because its
`tcap.h` declares termcap functions with K&R empty parens (`extern int
tgetnum ();`), which under C23 means "takes no arguments" -- so every
call site is a hard constraint violation, not a suppressible warning.
Bumped to **readline 8.3** today (upstream `tcap.h` has proper
prototypes); that's what unblocked this build.

What this run says about static vs the same-version dynamic baseline
(both 3.13.13, both built against the same OpenSSL 3.5.6 / SQLite
3.51.2 / zlib 1.3.2 pins, so the comparison isolates link mode rather
than library vintage):

- **Static is ~16% faster geomean across the suite, with the bigger
  wins on rows that benefit from cross-module inlining.** The pure
  eval-loop rows (`fib_iter`, `arith_loop`, `listcomp`, `dictops`,
  `attr_access`, `func_call`) sit at 1.08x-1.16x -- consistent with
  what whole-program LTO across the bytecode interpreter is expected
  to buy. The headline wins are `str_format` 1.31x, `json_roundtrip`
  1.30x, and `except_path` 1.24x -- exactly where the static build
  gets to inline C-level helpers (the PEP-3101 formatter, the JSON
  decoder's recursive descent, the error-restore path) through the
  interpreter dispatch boundary that `libpython3.13.so` would
  otherwise hide.
- **System python (Alpine 3.12.13) is consistently behind even the
  dynamic 3.13.13 build, and dramatically so on those same three rows:
  `str_format` 1.67x, `except_path` 1.48x, `json_roundtrip` 1.41x.**
  Two free variables here (version 3.12 vs 3.13, and compiler flags),
  so it's not a clean experiment. But the *direction* matches the
  static-vs-dynamic delta exactly, which makes "Alpine's aarch64
  python is missing the compiler-flag love (PGO? LTO? both?)" a more
  parsimonious explanation than "3.12 has genuinely slower formatter
  / exception / decoder code". The cheap probe is `python3 -c 'import
  sysconfig;
  print(sysconfig.get_config_var("PY_CORE_CFLAGS"))'` on
  `/usr/bin/python3` -- if PGO is absent, that's the gap.
- **Startup is dominated by the stdlib import path.** `bare` and
  `sysmod` are within 2 ms across all three interpreters (1.04x and
  1.13x dynamic/static respectively); the 1.20x spread only opens at
  `stdlib` (24.74 vs 29.58 ms). Consistent with the frozen-modules +
  no-`dlopen` story: the dynamic build pays per-`.so` to map and run
  each `PyInit_*`. The system python lands closer to static on the
  bare cases (1.03x, 1.04x), which is plausible if its stdlib's `.pyc`
  cache happens to be warm in the container's page cache from previous
  test runs -- worth re-running with a `vmtouch -e` between
  interpreters to check.

Side note: this is an Azure aarch64 VM whose kernel exposes only `CPU
implementer` / `CPU part` (raw MIDR_EL1 fields), not the `model name`
that x86 boxes do. `benchmark/run.sh:cpu_model()` was updated in this
same change to decode the `(implementer, part)` pair, which is how the
Environment block above prints `ARM Neoverse-N1` and how the table of
caches makes sense (L1d/L1i 64K, L2 1024K, L3 32768K is what Neoverse-N1
is expected to expose). The cpu line in this report was patched in by
hand to match the post-fix output; future reports will populate it
directly.

Concrete next steps:

1. `python3 -c 'import sysconfig;
   print(sysconfig.get_config_var("PY_CORE_CFLAGS"))'` on
   `/usr/bin/python3` to confirm whether Alpine's aarch64 package
   ships PGO + LTO. That answers most of the system/static gap in one
   command.
2. `perf stat -e instructions,cycles,branch-misses,
   L1-icache-load-misses` on the static and dynamic binaries running
   the `json_roundtrip` body -- attributes the 1.30x win to inlining
   (lower instruction count) vs icache locality (lower L1i misses) vs
   better branch prediction. Single cleanest experiment to characterise
   where the LTO gains come from on a C-extension-heavy row.
