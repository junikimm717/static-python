## Analysis

First apples-to-apples PGO+LTO x86_64 report. Same host as the
[1924Z](./2026-05-17T1924Z_x86_64.md) run (Ryzen AI 9 HX 370, kernel
6.18, container), same gcc 15.1.0, same library versions -- the only
intentional change is **PGO now on for both** static and dynamic. (The
unblock for static was the musl-fma workaround documented in
`BUGREPORT.md`: `-i test_fma_zero_result` added to `PROFILE_TASK` in
both `Makefile` and `benchmark/dynamic-build.sh`.) This run resolves
the "rebench x86_64 with the current static recipe" item from the
[cliffs-notes README](./README.md) and gives us the first non-aarch64
data point for the canonical PGO+LTO numbers.

Three natural diff targets:

- [`2026-05-17T1924Z_x86_64.md`](./2026-05-17T1924Z_x86_64.md) -- same
  host, pre-PGO. Tells us what PGO bought on x86_64 specifically.
- [`2026-05-17T2146Z_aarch64.md`](./2026-05-17T2146Z_aarch64.md) -- M4
  Pro / OrbStack, PGO+LTO.
- [`2026-05-17T2359Z_aarch64.md`](./2026-05-17T2359Z_aarch64.md) --
  Azure Neoverse-N1, PGO+LTO, real PMU.

With this report there are now **three** PGO+LTO data points across
three µarchs (M4 Pro Firestorm P-core, ARM Neoverse-N1, AMD Zen 5),
so the canonical-recipe-vs-anything claims have to be sourced from
that 3-host matrix rather than from any single report.

### The json_roundtrip regression has vanished

This is the big one. The 1924Z analysis ended with:

> "static + `-O3 -flto` on the JSON decoder path takes ~25-30% longer than
> the dynamic build's per-TU compilation does, on any x86_64 host we've
> measured." That is the next thing to actually instrument.

**Hypothesis falsified.** Turning PGO on for both flipped `dynamic/static`
from 0.75x to 1.19x. The static absolute time on this row halved (8.94 us
-> 4.10 us); the dynamic absolute time also halved (6.70 us -> 4.88 us)
but less so. So the 1924Z "static is slower on the JSON decoder" claim
was really "PGO matters more for the static build's whole-program
`-flto-partition=none` recipe than it does for the dynamic build's
per-extension build". The cliffs-notes "What is mixed or wrong" table
can lose the `json_roundtrip` x86_64 row entirely; this report's value
(1.19x) lines up with the M4 Pro PGO+LTO report's value (1.22x).

### libpython-internal rows compressed, C-extension-boundary rows widened

This is the same shape the M4 Pro 2146Z aarch64 run showed: PGO closes
the dynamic-side gap on rows whose hot path is mostly libpython
internals (`fib_iter`, `listcomp`, `dictops`, `except_path`, `func_call`,
`regex_match`), because that's exactly where the upstream `--with-lto`
+ PGO recipe was already doing most of its work for the dynamic
interpreter. The rows where static is still pulling ahead are the ones
that *cross* an extension boundary: `arith_loop` (bytecode int math is
internal but the inner mul/add path hits `_PyLong`), `str_format`
(`_string.formatter_parser`), `json_roundtrip` (`_json` decoder), and
`attr_access` (`LOAD_ATTR` specialization which hits `_collections_abc`
on first dispatch). On those rows the static binary's whole-program LTO
across `libpython + _json + _string + ...` extends inlining visibility
into the C extension code that the dynamic build's libpython-internal
`-flto` cannot reach.

### `listcomp` sits in the middle on Zen 5, doesn't cleanly fit the L1i story

2359Z's leading hypothesis for the M4-Pro-vs-N1 `listcomp` gap (1.27x
vs 1.07x) was L1i cache thrashing under aggressive
`-flto-partition=none` inlining -- M4 Pro's 192 KiB L1i fits the
inlined hot body, N1's 64 KiB L1i doesn't. The Zen 5 number is a
useful test of that hypothesis, because Zen 5's L1i is **32 KiB** --
*smaller* than N1's. If L1i sizing were the dominant variable, Zen 5
should land *below* N1 on `listcomp`. It doesn't:

| host         | L1i   | listcomp d/s | static abs |
|---           |---:   |---:          |---:        |
| M4 Pro       | 192K  | 1.27x        | 15.6 ns    |
| Zen 5 (here) | 32K   | **1.18x**    | 27.7 ns    |
| Neoverse-N1  | 64K   | 1.07x        | 62.8 ns    |

So the L1i-thrashing hypothesis at most explains *part* of the
M4-vs-N1 spread; something else is helping Zen 5 hold a 1.18x win on
a much smaller L1i. Plausible candidates: Zen 5's 6.75K-entry op
cache (effectively a second-level decoded-instruction store that
sits in front of the L1i for hot loops) sidesteps the icache-miss
cost for the static binary's larger hot path; or Zen 5's branch
predictor handles the over-inlined recursive structure better than
Neoverse-N1's. A `perf stat -e L1-icache-load-misses,
L1-icache-loads,icache_op_cache.hit` invocation on `listcomp` on the
Zen 5 box, paired with the same on N1, would settle which of those
two stories holds.

### Startup geomean collapsed -- and static now loses on `stdlib`

| scenario | 1924Z dyn/static | 0000Z dyn/static |
|---       |---:              |---:              |
| bare     | 1.12x            | 1.06x            |
| sysmod   | 1.22x            | 1.08x            |
| stdlib   | 1.18x            | **0.89x**        |
| **geomean** | **1.17x**     | **1.01x**        |

PGO of the dynamic baseline competed away most of the startup
advantage, and on the `stdlib` scenario (which exercises a deeper
import graph than `bare` or `sysmod`) the dynamic binary now spawns
*faster* than the static one. The pre-PGO claim that "static binary
beats dynamic on every startup scenario" no longer holds on x86_64.

Why does `stdlib` flip? The static binary actually *shrank* over
1924Z (15.1M -> 13.9M), probably because PGO + `--gc-sections` let
the linker drop cold import-time code that `-O3 -flto` alone couldn't
prove dead. The dynamic binary's `libpython3.13.so` is essentially
unchanged in size (30.7M -> 30.9M). So a "static binary got bigger
and is paying more page-fault tax" story doesn't fit. More likely the
dynamic side got *cheaper* at imports under PGO: PGO has a known
disproportionate effect on the marshal-and-eval frozen-module path,
which is exactly what dominates `stdlib`'s 22 ms. Worth a `strace -e
openat,read,mmap -c` comparison if we end up caring.

### Next probes

1. **`perf stat -e cycles,instructions,branch-misses,iTLB-load-misses,
   L1-icache-load-misses` on `fib_recursive`, on both N1 and Zen 5,
   both binaries.** 2359Z queued this for N1; we now have two real-PMU
   hosts with the same regression at different magnitudes (0.92x vs
   0.89x), which is *better* than one because cross-host agreement on
   the offending counter narrows the mechanism. If static retires
   more instructions than dynamic on the same `fib(20)`-in-a-loop
   body, LTO is over-inlining the eval-loop's recursive return path.
   If it retires the same instructions slower, the eval-loop's hot
   path is spilling out of L1i / iTLB. The promoted-from-2359Z item.
2. **`perf stat -e L1-icache-load-misses,L1-icache-loads` on
   `listcomp` on N1 and Zen 5.** Tests the 2359Z L1i-thrashing
   hypothesis with the help of this report's "Zen 5 holds 1.18x on a
   smaller L1i than N1" counter-evidence. If N1 has a measurably
   higher icache miss rate on `listcomp` than both M4 Pro and Zen 5,
   the L1i story is right and Zen 5's saving grace is something else
   (op cache being the obvious candidate). If miss rates are
   comparable on all three, the L1i hypothesis is wrong.
3. **`stdlib` startup regression triage on Zen 5.** `strace -e
   trace=openat,read,mmap,close -c` on `python -c 'import sys'` vs
   `python -c 'import sys, ...stdlib...'` for both binaries. The
   `stdlib 0.89x` on Zen 5 is one data point and is x86_64-specific
   (M4 Pro and N1 both hold 1.13x on the same scenario), so this is
   probably either a Ryzen-host-specific scheduling artefact or
   something the dynamic-side PGO captured about the import path on
   musl-x86_64 that it didn't capture on musl-aarch64. Easy 1-hour
   answer.
4. **`DYNAMIC_USE_INTREE=1`** -- still the #1 entry in the cliffs-notes
   "What is missing" list. With this run, three of the four widest
   surviving wins (`str_format`, `attr_access`, `json_roundtrip`,
   `arith_loop`) are exactly the C-extension-boundary rows that
   DYNAMIC_USE_INTREE would isolate. If those drop from 1.19-1.52x
   toward parity against an LTO'd-Alpine-libs dynamic build, "the
   static win is mostly that Alpine's libs aren't LTO'd" becomes the
   published answer.
5. **Same-host re-bench in 24h** to band the noise. CPython
   microbenches against `-ffunction-sections --gc-sections` static
   binaries are known to be sensitive to relink layout; the
   `fib_recursive 0.89x` and `stdlib 0.89x` headline numbers should
   be confirmed before they get used as priors for (1)-(3).
