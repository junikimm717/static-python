## Analysis

First run on the `juni/patch-for-ldplugin` branch. The intentional
change is on the **static** side only: the musl-cross gcc 15.1.0 host
toolchain is now built **without** `--disable-shared`, so it ships
`liblto_plugin.so`; the host gcc itself is dynamically linked and
relocatability is restored by the static-musl launcher shim in
`cross-make/wrapper.c` + `cross-make/post-install.sh`. With that
plugin available, the Python build flags in `configure-wrapper.sh`
gained `-fuse-linker-plugin -fno-fat-lto-objects` on top of the
existing `-O3 -flto -flto-partition=none`. Dynamic and system rows
are unchanged in build recipe vs prior reports.

### Two diff targets, and why neither is clean

This is a Cascade Lake (Azure) run, so the **same-host** prior is
[`2026-05-17T1818Z_x86_64.md`](./2026-05-17T1818Z_x86_64.md) -- but
that one is pre-PGO on both sides, so the diff bundles "PGO on both"
with "linker plugin on static". The **same-recipe** prior on x86_64 is
[`2026-05-18T0000Z_x86_64.md`](./2026-05-18T0000Z_x86_64.md) on Ryzen
AI 9 HX 370 (Zen 5) -- so the diff bundles "linker plugin on" with
"µarch change Zen 5 -> Cascade Lake". Treat both as confounded; the
honest read is the intersection.

| host                       | era                      | geomean CPU | startup |
|---                         |---                       |---:         |---:     |
| 1818Z Cascade Lake         | pre-PGO, no LP           | 1.17x       | 1.13x   |
| 0000Z Zen 5                | PGO+LTO, no LP           | 1.18x       | 1.01x   |
| **2322Z Cascade Lake (this)** | **PGO+LTO + LP**     | **1.20x**   | **1.11x** |

CPU geomean moves 1.17x -> 1.20x same-host (adding PGO + LP together)
and 1.18x -> 1.20x cross-host at the same recipe sans LP. The
cross-host PGO+LTO range across previously-banked hosts is **1.07x
(N1) - 1.18x (Zen 5)** -- this run lands one tick above the top of
that range, but the µarch change alone is plenty of slack to account
for it. No headline win attributable to the linker plugin.

The headline hypothesis we set up for `-fuse-linker-plugin` was that
it should especially help rows that bottom out in cross-archive
symbol resolution against the in-tree static libs -- `json_roundtrip`
(libc + `_json`), `regex_match` (`_sre`), `str_format` (`_string`),
and anything touching openssl/sqlite/libffi/libz. **That story does
not survive contact with the numbers.** `json_roundtrip` is flat at
1.20x, `str_format` actually *narrowed* by 9 pp, `regex_match` moved
+4 pp which is barely above the noise floor. The rows that widened
the most (`listcomp` +5, `dictops` +6, `fib_iter` +5, `except_path`
+4) are all libpython-internal hot paths, not cross-archive ones --
the opposite of where the plugin was supposed to help. The most
likely read is that what we are seeing is µarch noise (Cascade Lake's
wider issue width + larger L2 vs Zen 5's op cache) rather than a
plugin signal, and the plugin's contribution is buried under it.

| host           | core        | fib_recursive |

| M4 Pro 2146Z   | Firestorm   | 0.78x         |

| N1 2359Z       | Neoverse-N1 | 0.92x         |

`-fuse-linker-plugin` did not move the `fib_recursive` regression.
This now has four PGO+LTO data points across four µarchs spanning
both ISAs -- the regression is a property of the recipe, and the
[0024Z N1 perf follow-up](./2026-05-18T0024Z_aarch64_perf.md) already
attributed it to branch-mispredict + back-end stalls inside the
computed-goto dispatch in `_PyEval_EvalFrameDefault`, not L1i. The
linker plugin doesn't touch that mechanism, so this null result is
expected and the open `perf stat` follow-up on Zen 5 is still owed.

### `stdlib` startup did not regress here

0000Z on Zen 5 surprised everyone with `stdlib` dyn/static at
**0.89x** (dynamic spawned faster). This run lands 1.14x, right
where M4 Pro (1.13x) and N1 (1.13x) sit. That preserves the 0000Z
read: the 0.89x was a Zen-5-only artefact, not something the linker
plugin or any cross-arch change could have introduced. With this
report there are now three PGO+LTO `stdlib` data points at 1.13-1.14x
and one outlier at 0.89x; the outlier is the thing to investigate,
not the cluster.

### Static binary grew 0.9 MB same-host (15.1M -> 16.0M)

1818Z static was 15.1M; this run is 16.0M, with both built on the
same host. PGO + `--gc-sections` was the size-shrinking effect
behind 0000Z's 13.9M number on Zen 5; we still have all of that, so
the 0.9 MB growth must come from elsewhere. The two suspects are:
(1) PGO instrumentation training picked up extra hot paths the
no-PGO 1818Z linker had pruned, (2) `-fuse-linker-plugin` lets the
linker pull additional reachable code out of `.a`s that `-flto`
without the plugin had been silently dropping. The second is what
the plugin is *for* and is consistent with the absent CPU geomean
win -- if the plugin is just pulling in more reachable code without
opening new inlining frontiers, you get binary growth and ~no perf
delta, which is what we see.

### Next probes

1. **Same-host LP off/on differential.** The cleanest experiment to
   actually attribute anything to `-fuse-linker-plugin` is to add a
   `STATIC_NO_LINKER_PLUGIN=1` env knob to `configure-wrapper.sh`
   that drops `-fuse-linker-plugin -fno-fat-lto-objects` from both
   `CFLAGS` and `LDFLAGS`, rebuild, and re-bench on **this Cascade
   Lake host** so neither µarch nor PGO can muddy the diff. Without
   that we cannot say the geomean move from 1.17x to 1.20x is the
   plugin -- almost all of it is PGO.
2. **Binary size attribution.** `size -A
   python-static-x86_64-linux-musl/bin/python3.13` and
   `nm --size-sort -S` against the 1818Z and 0000Z binaries (or, if
   those are gone, against a `STATIC_NO_LINKER_PLUGIN=1` rebuild)
   would tell us whether the +0.9 MB is `.text` (plugin pulling in
   more code from archives) or `.data`/`.rodata`/PGO counters. The
   plugin-vs-no-plugin diff is the interesting one; a `comm -23
   <(nm plugin.bin | sort) <(nm noplugin.bin | sort)` lists the
   symbols the plugin made reachable.
3. **`perf stat -e cycles,instructions,branch-misses,
   iTLB-load-misses,L1-icache-load-misses` on `fib_recursive` and
   `json_roundtrip`**, this host, both static and dynamic. The N1
   follow-up settled `fib_recursive` for one µarch; Cascade Lake
   would be a second real-PMU data point, and would also let us
   confirm whether the flat `json_roundtrip` 1.20x is really
   front-end-bound (which the plugin can't help) or back-end-bound
   (which the plugin should help via better inlining). This was
   already owed from 0000Z under #1 in the cliffs-notes "Open
   experiments" list; it now has two motivating reports.
4. **Re-bench Cascade Lake without changing anything, in 24h.**
   Most of the per-row moves vs 0000Z are µarch-attributable rather
   than plugin-attributable, but the geomean lands one tick above
   the established cross-host band (1.07-1.18 -> 1.20). A repeat
   run pins how much of that one tick is noise before any of (1)-(3)
   gets prioritised.
